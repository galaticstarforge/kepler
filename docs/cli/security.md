# Security

## Credentials

The CLI never stores or transmits AWS credentials. It relies entirely on the AWS SDK's standard credential resolution chain. Credentials flow from the environment to AWS SDK calls and nowhere else. There is no credential caching, no credential forwarding to the deployed instance, and no credential serialization to disk.

---

## State Bucket Access

The state bucket's bucket policy restricts access to the owning AWS account. Cross-account access is out of scope for v1 and would require explicit bucket policy additions.

The CLI does not assume any particular IAM identity. Whatever credentials are available are what gets used. A CLI user may have full admin permissions or the scoped `KeplerUserAccess` policy. Both work, with different capabilities on destructive operations.

---

## CloudFormation Permissions

`kepler deploy` requires permissions that go beyond `KeplerUserAccess`: CloudFormation write, IAM role creation, EC2/VPC provisioning, S3 bucket creation, and Bedrock KB creation. Users performing deployments need a more permissive IAM policy, typically `PowerUserAccess` plus IAM, or a custom deploy policy. This policy is not generated by `kepler iam-policy`, which only generates the read/access policy for day-to-day users.

This is a standard AWS pattern: separation of concerns between infrastructure provisioning and day-to-day operation. The people who deploy the stack and the people who use the stack do not need the same permissions.

---

## No Telemetry

The CLI does not send telemetry. No usage data, no crash reports, no version checks other than the explicit `kepler upgrade` command. The user's AWS credentials are used only to talk to their AWS account. There are no hidden network calls to any external service.

---

## Audit

Every CLI operation that modifies state is logged through three channels:

- **AWS CloudTrail** logs every API call made by the CLI, with the IAM principal, source IP, and timestamp.
- **The state bucket's `history/` prefix** receives a YAML record for every deployment and plugin operation, with timestamp, IAM principal (from `STS:GetCallerIdentity`), operation type, and outcome.
- **SSM session records** for tunnels and shells appear in CloudTrail with full session metadata.

No additional audit infrastructure is required. The CLI piggybacks on AWS's native audit surfaces.

---

## Network Attack Surface

The deployed stack's attack surface at the network layer is zero inbound. There are no public IPs, no inbound security group rules, and no public DNS entries. The only path to the instance is via IAM-authenticated SSM. An attacker who compromises a user's AWS credentials can reach the instance. An attacker without AWS credentials cannot, regardless of network position.

This is the principal benefit of the SSM-over-SSH decision. It's not just simpler credential management. It's a genuinely smaller attack surface.
