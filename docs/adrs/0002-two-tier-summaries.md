# ADR-0002: Two-Tier Provisional and Canonical Summaries

**Status:** Accepted  
**Date:** 2026-04-16

---

## Context

Generating a full, high-quality summary for every symbol in a codebase is expensive. A codebase with 50,000 symbols, described at 4,000 input tokens and 500 output tokens per symbol, would cost several thousand dollars per full pass. That is impractical as a baseline.

At the same time, having no summary for frequently-queried symbols produces a poor MCP consumer experience: the developer asks "what does `processPayment` do?" and gets back structural metadata but no natural-language description.

Three approaches:

1. **Uniform coverage.** Summarize every symbol at the same quality. Cost is prohibitive at scale. Quality is identical regardless of symbol importance, which wastes budget on symbols never queried.
2. **Top-N only.** Summarize only the top N symbols by pageRank. Simple. But N is hard to set correctly across different codebases, and this leaves entire subsystems with no summaries if they are internally important but externally low-traffic.
3. **Two-tier: provisional and canonical.** Every symbol eventually gets a summary, but the summary quality depends on the symbol's importance. Provisional summaries are cheap and fast. Canonical summaries are expensive and high-quality. Provisional summaries can be upgraded to canonical without re-processing the community.

The two-tier approach allows full coverage (all symbols have at least a provisional summary) while concentrating canonical-model budget on high-importance symbols.

---

## Decision

I use two summary tiers:

**Provisional:** generated by the navigation model (cheap/fast). Contains `purpose` (one sentence), `semanticTags` (3-5 tags), and `tier: 'provisional'`. Suitable for bulk coverage and for giving MCP consumers minimal orientation. Generated from docstring, symbol properties, and structural inference when source reading is skipped.

**Canonical:** generated by the summary model (high quality). Contains the full `SymbolSummary` schema: `purpose`, `details`, `sideEffects`, `semanticTags`, `examplesFromTests`, plus coverage flags. Generated from full source read, caller/callee context, community context, and docstring. Canonical summaries go through the graph validation loop.

Promotion rules:
- Symbols with `pageRank >= 0.05`, `isPublicApi = true`, or `communityRole = 'bridge'` are canonical candidates on first pass.
- All other symbols receive provisional summaries first. They are eligible for promotion during upgrade passes.
- The upgrade pass runs after the full first pass and processes remaining provisional summaries in priority order within budget.

The `tier` field on `SymbolSummary` is always returned to MCP consumers. Consumers that need high-confidence descriptions can filter for `tier: 'canonical'`.

---

## Consequences

### What This Enables

- Full symbol coverage from the first pass, even though only high-importance symbols get canonical-quality summaries initially.
- The upgrade pass is structured as a background, low-priority job. It does not block the summarization subsystem from serving queries.
- MCP consumers know the quality of what they are reading. A consumer building a critical code review tool can query only canonical summaries. A consumer building a discovery tool can use provisionals for breadth.
- Cost scales with codebase complexity, not codebase size. A large codebase with many trivial utility functions does not cost much more than a small codebase with the same number of complex core symbols.

### What This Costs

- Two summary schemas means two prompt templates, two validation paths, and two sets of coverage metrics. More moving parts.
- Provisional summaries generated from docstrings and structural inference can be wrong. If a symbol has a misleading docstring and low importance, it will have a misleading provisional summary indefinitely. Mitigation: the `coverageFlags` field records whether a docstring was available, so consumers can determine how the provisional was generated.
- The tier boundary (what makes something a canonical candidate) is a configuration threshold. Wrong thresholds produce either too much cost (threshold too low) or too little quality (threshold too high). The defaults are empirically derived from typical open-source codebase distributions; adjustment is expected for specific projects.

### Failure Modes

- If the navigation model produces negligibly informative provisional summaries (e.g., "This is a function that does things"), the provisional tier provides little value. Mitigation: the agent prompt template for provisionals requires structured output with explicit failure when it cannot produce at least a one-sentence characterization.
- If many symbols are at exactly the tier boundary and the upgrade pass prioritizes some but not others, there can be apparent inconsistency in which symbols have canonical summaries. This is expected behavior; the priority ordering is documented and reproducible.

---

## Alternatives Considered

**Uniform coverage at canonical quality:** too expensive. A codebase of 50,000 symbols at $0.02 per symbol would cost $1,000 for the first pass, and that repeats whenever significant code changes occur.

**Top-N only:** leaves large code areas with no semantic coverage. A developer exploring a module that happens to have low pageRank gets nothing useful from the MCP server. The two-tier approach treats low-importance symbols as first-class citizens; they just get lighter treatment.

**No summaries, only structural metadata:** acceptable for some use cases but fails the MCP consumer experience test for "what does this function do?" questions, which are the most common query type.
